## 🔄 关键API 调用流程

### 创建 → 加入 → 填写 → 协商 → 选择

```
用户 A (队长)              用户 B, C, D (成员)
   |                              |
   v                              v
createTeam()          ←---- 收到邀请链接 ----→ joinTeamByCode()
   ↓                                           ↓
生成邀请码和链接        ←--- 搜索或发现 ---→ searchPublicTeams()
   ↓                                    ↓
分享给朋友              ←--------- ✅ 加入 --------→ 进入 PreferenceFilling
   |                                    |
   ├─────── updateMemberPreferences ────┤
   |         (mood, difficulty, etc.)   |
   |                                    |
   └─────── 检查 areAllCompleted ────────┤
            ↓                            ↓
         [启用按钮]            [所有人都完成了]
            ↓
  performGroupNegotiation()
       |
       ├─ 收集所有偏好
       ├─ 调用 Gemini AI
       ├─ 获得综合分析
       ├─ findMatchingRoutes()
       └─ 返回 5 条推荐
       
            ↓
  [显示推荐给所有队员]
       ↓
  selectGroupRoute()
       ↓
  [跳转到 CompanionView 开始登山]
```

## 🚀 立即可用的代码

### 前端集成示例

```typescript
// 1. 导入
import * as teamService from '../services/teamService';
import * as groupService from '../services/groupNegotiationService';

// 2. 创建团队
const team = await teamService.createTeam(
  'Dragon Peak Club',
  'Weekend hiking adventure',
  true,
  4
);
console.log('邀请链接:', team.invite_link);

// 3. 获取团队信息
const teamData = await teamService.getTeamWithMembers(teamId);
console.log(`完成度: ${teamData.completionPercentage}%`);

// 4. 更新我的偏好
await teamService.updateMemberPreferences(
  teamId, userId,
  'Peaceful', 'easy', 'Quiet forest',
  300, 20
);

// 5. 检查是否可以协商
const ready = await teamService.areAllPreferencesCompleted(teamId);

// 6. 启动协商
if (ready && isLeader) {
  const result = await groupService.performGroupNegotiation(teamId, userId);
  
  // result 包含:
  // - synthesizedPreferences (AI 分析)
  // - recommendedRoutes (5 条推荐)
  // - autoGeneratedTitle (自动标题)
  // - negotiationId (用于保存选择)
}

// 7. 选择路线
await groupService.selectGroupRoute(teamId, selectedRouteId, negotiationId);
//→ 自动跳转到 CompanionView
```

## 💡 关键技术亮点

### 1. AI 驱动的社交协商 ✨
- 不是简单的平均值，而是真实的"妥协"分析
- Gemini 理解背景，提供人性化的建议
- 队员看到"为什么"被选中了这条路

### 2. Segment 系统的创新使用
- 将 segment 系统从 solo 扩展到 group
- 同一个 segment 库为两种模式服务
- 路由推荐基于综合但不失个性的偏好

### 3. 仔细的权限设计
- RLS 政策确保隐私
- 队长权限与成员权限分离
- 防止误操作 (弹出确认对话框)

### 4. 基于行为的学习
- 记录所有协商结果和选择
- `team_negotiation_history` 可用于:
  - 调整 AI 提示词
  - 改进推荐算法
  - 用户研究

## 🔍 数据结构亮点

### 邀请码设计
- **6 位随机码** (ABC123)
- 足够短: 易于分享、记忆、输入
- 足够长: 36^6 ≈ 2.1B 种 (足以支持百万级团队)
- **自动生成**: 触发器在 INSERT 时自动创建

### 综合偏好存储
```json
// 存储的内容:
{
  synthesized_mood: "Peaceful + Scenic",
  synthesized_difficulty: "medium",
  synthesized_condition: "Forest hike with photo spots",
  
  // + analysis 和 considerations 供 AI 学习
}
```

### 推荐路由的记录
```json
{
  recommended_routes: [
    {
      id: "route_123",
      name: "Forest & Wildlife Trail",
      score: 87,
      matchReasons: ["High tag match", "Perfect difficulty"]
    },
    ...
  ],
  selected_route_id: "route_123"  // 用户最终选择
}
```

## 📞 支持和常见问题

### Q: 一个团队最多支持多少成员?
A: 当前设置 max_team_size = 50。可以修改数据库约束。

### Q: 邀请码过期吗?
A: 当前设计不过期。可选:在 team_invite_links 表中添加 expires_at。

### Q: 可以允许用户修改偏好后自动重新协商吗?
A: 可以。添加一个触发器在 updateMemberPreferences 后发送通知。

### Q: 如何处理网络离线?
A: 使用 Service Worker 缓存，恢复后同步。

### Q: 协商结果可以共享吗?
A: 生成一个团队的 "public page" URL，分享路线和推荐。
